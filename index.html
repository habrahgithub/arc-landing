ARC XT — Audit Ready Core

ARC XT is a deterministic, tamper-evident audit layer that operates at the moment code changes are made. It enforces governed code enforcement before decisions reach PR, CI, or production.

What ARC adds
  • Every critical decision has a verifiable origin.
  • Every change is linked to intent — not just diff.
  • Every action is provable after the fact.

Problem

AI speeds output. It breaks accountability.

PR review shows diffs, not decisions. CI tests outcomes, not intent. When code is generated faster than it can be reviewed, “why” disappears.

Today, accountability starts at PR or CI. By then, the decision is already lost.

As AI becomes the default way code is written, auditability becomes a requirement — not a feature.

If you can’t reliably answer who decided what, when, and why before PR, you don’t have an audit layer.

Missing layer

Why a new layer is required

AI changed how code is created.

PR and CI systems evaluate changes after decisions are already made.

Review can inspect code, but it cannot recover decision origin once it is missing.

A distinct layer must exist at the moment of change, before PR and CI.

How it works

ARC XT is the first implementation of the ARC audit layer in VS Code.

ARC XT is a local-first VS Code extension that records and verifies critical decisions before a change reaches PR/CI.

01

You make a critical change

ARC activates where it matters: auth, schema, config, and other high-risk edits.

02

ARC records the decision + proof

Provide a Change ID to link the change to a plan-backed blueprint artifact.

03

ARC writes tamper-evident evidence

Audit events are chained and verifiable so integrity can be checked and drift detected.

ARC XT sits below AI tools and before Git/PR/CI — inside the moment of change.

Heuristic Evaluation

The Decision Engine

Four enforcement levels mapped to your governed workspace.

ALLOW

Low-risk changes such as UI components or tests. Save proceeds normally.

WARN

Medium-risk changes like config or schema edits. Requires operator acknowledgment.

REQUIRE_PLAN

High-risk code logic. Requires linking a verifiable Blueprint Proof artifact.

BLOCK

Critical-risk policy violations. Save is completely blocked and fail-closed.

Proof

What you get after a critical change

Decision record

Read-only descriptive review captured at save-time — not reconstructed later.

Blueprint proof

Instructional guidance linked to a Change ID, capturing bounded authorization.

Hash-chain integrity

Tamper-evident .arc/audit.jsonl audit entries you can cryptographically verify.

Differentiation
